## 1. 项目定位与目标

### 1.1 这个 App 是做什么的
WhatShallWeEat 是一个“吃什么决策工具”，解决的是日常高频但低效的决策问题：
- 今天吃什么？
- 已经想了很多候选菜，怎么快速选一个？
- 某天我到底吃了什么，后续如何回看？

### 1.2 当前产品价值
- **降低决策疲劳**：通过候选池 + 抽奖，减少纠结时间。
- **记录饮食轨迹**：按日期保存菜单，在日历中查看。
- **轻量可离线**：纯本地存储，无后端依赖，打开即用。

### 1.3 目标用户（当前版本）
- 个人/情侣/小家庭日常做饭或点餐决策。
- 不想维护复杂饮食系统，只想快速做决定的人群。

## 2. 当前技术实现（微信小程序）

### 2.1 技术栈
- 微信小程序原生：`WXML + WXSS + JavaScript`
- 无后端服务
- 数据持久化：`wx.setStorageSync / wx.getStorageSync`

### 2.2 页面结构（Tab）
- `pages/order/index`：菜单主页面（核心流程）
- `pages/calendar/index`：日历查看与按日编辑
- `pages/mine/index`：我的（当前为占位页）

### 2.3 核心组件
- `dish-card`：菜品卡片（加入今日菜单、加入/移出候选池）
- `category-chips`：分类筛选
- `search-bar`：搜索
- `meal-calendar`：日历展示与日期切换
- `custom-tab-bar`：底部导航

### 2.4 数据源
- 静态菜品数据：`data/dishes.js`
- 菜品字段已标准化到 `dishId/name/categoryId/emoji/img/imgFileId/xhsLink`，并兼容旧字段。

## 3. 用户怎么使用这个 App（按真实流程）

### 3.1 菜单页（日常主流程）
1. 打开小程序进入“菜单”Tab。
2. 用左侧分类筛选（鸡肉、牛肉、猪肉、鸭肉、鱼肉、虾类、蔬菜、特殊）。
3. 可输入关键词搜索（按菜名/描述过滤）。
4. 点击菜卡按钮将菜加入“今日菜单”。
5. 对暂时拿不准的菜，可点“容我想想”加入候选池（THINK_POOL）。
6. 点击底部“查看已选”打开今日菜单弹窗。
7. 在弹窗中点击“抽奖”，从候选池随机抽取 N 道菜。
8. 抽奖“开始/再抽”仅预览；点“确定”才会写入当天菜单。

### 3.2 日历页（回看与补录）
1. 切换到“日历”Tab。
2. 查看每个日期下方 emoji 标记（代表当天选过的菜）。
3. 选中某天后，查看该日期已选菜品。
4. 点击“编辑”进入编辑态：
   - 可移除该日期已选菜
   - 可从全部菜品中“补录添加”到当天

### 3.3 我的页
- 当前版本主要为结构占位，未承载复杂业务。

## 5. 当前数据模型与存储键（非常关键，迁移必读）

### 5.1 核心存储键
- `THINK_POOL`：全局候选池（`DishItem[]`）
- `TODAY_MENU_YYYY-MM-DD`：某天菜单（`DishItem[]`）
- `LAST_LOTTERY_RESULT_YYYY-MM-DD`：某天最近确认抽奖结果（`DishItem[]`）
- `meal_calendar_marks_v1`：日历标记（`{ [date]: emoji[] }`）

### 5.2 兼容旧键
- `MEAL_PLAN_BY_DATE`：旧的日期 -> dishId[]
- `MEAL_PENDING_BY_DATE`：旧的日期 -> dishId[]

### 5.3 DishItem 标准结构
```ts
{
  dishId: string,
  id: string,           // 兼容别名
  name: string,
  categoryId: string,
  category: string,     // 兼容别名
  emoji?: string,
  img?: string,
  imgFileId?: string,
  xhsLink?: string
}
```

### 5.4 抽奖算法
- 使用 Fisher-Yates 风格洗牌后切片
- 具备边界保护：抽取数量会 clamp 到 `[1, poolSize]`

## 6. 已知边界与技术债（Swift 重构时可直接修复）

1. **无账号体系**：仅设备本地存储，无法跨设备同步。
2. **无云端数据**：删除小程序或更换设备会丢失历史。
3. **候选池是全局**：目前不是按日期/餐别维度管理。
4. **“我的”页能力不足**：缺少统计、偏好、设置中心。
5. **菜谱链接能力未形成完整闭环**：字段已预留，但当前主要逻辑仍围绕菜单决策。
6. **测试体系较轻**：以手动验证为主，缺少自动化用例。

## 7. Swift 迁移建议（分层设计）

### 7.1 架构建议
推荐采用：
- UI：`SwiftUI`
- 状态管理：`ObservableObject + @StateObject/@EnvironmentObject`
- 数据层：`Repository + UseCase`
- 本地存储：`SwiftData`（iOS 17+）或 `Core Data`（兼容更多版本）

### 7.2 模块拆分建议
1. `DishCatalogModule`
   - 菜品列表、分类、搜索
2. `MealPlanModule`
   - 今日菜单管理、按日增删
3. `ThinkPoolModule`
   - 候选池管理
4. `LotteryModule`
   - 抽奖策略、预览与确认
5. `CalendarModule`
   - 日历标记、按日回看与编辑
6. `Settings/ProfileModule`
   - 用户偏好、备份恢复、导出导入

### 7.3 数据迁移策略
- 第 1 版 Swift 可以先延续当前字段模型，降低迁移风险。
- 对于旧键兼容逻辑，在 Swift 中只保留“导入脚本”能力，应用运行态只维护新模型。
- 建议加入 `schemaVersion`，每次模型变动做可回放迁移。

## 9. 推荐迭代路线（你后续如果继续推进）
